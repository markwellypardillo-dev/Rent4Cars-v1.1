1. System Overview
1.1 Purpose of the System

Rent4Cars is a comprehensive, full-stack web application designed to streamline the car rental process while providing robust post-booking support. The platform allows users to browse an up-to-date fleet of vehicles, select service hubs for pickup and drop-off, and communicate in real-time with logistics coordinators and AI-augmented expert mechanics for on-the-road troubleshooting.

The platform addresses common pain points in traditional car rental services by bridging the gap between simple vehicle reservation and active, on-the-road technical support. By utilizing a real-time WebSocket-powered Admin Live Chat and a localized XML database structure, Rent4Cars ensures that users have immediate, 24/7 access to human support and troubleshooting assistance, significantly enhancing passenger safety, operational efficiency, and overall customer satisfaction during their journeys.

Core Mission: Deliver an end-to-end car rental platform that blends a modern React frontend with robust XML-driven data management, backed by intelligent WebSockets for live human support and logistics.
Key Capabilities
  • Real-time vehicle availability tracking and robust booking flow
  • Admin Live Support powered by Socket.io for instant roadside assistance and diagnostics
  • Interactive live dual-way messaging gateway between users and admins
  • Location and Hub tracking for pick-up/drop-off networks
  • Structured XML data layer replacing traditional relational databases
  • XML-to-HTML translation directly on the backend via XSLT pipelines
  • High-performance SPA using React 19 and Vite

3. Feature Descriptions
3.1 Messaging System

The system features an active live messaging gateway via WebSockets. Users can contact Rent4Cars support to engage with a human logistics admin or mechanic in real time for booking discussions or car diagnostics. The backend intelligently routes messages between users and the admin portal. This chatbot interface sits persistently on the UI as a floating block, syncing live updates seamlessly.

3.2 XML Usage

To satisfy structural data requirements, the application relies entirely on XML as a localized data layer. Files such as fleet.xml, locations.xml, and messages.xml map out core entities. This ensures the data is strictly structured, portable, and easily parsable by secondary enterprise systems if required without needing SQL database infrastructures.

3.3 Parsing Methods

On the Node.js backend, the system utilizes the DOMParser from the @xmldom/xmldom library. When the backend receives a request (e.g., fetching fleet details or adding a message), it reads the physical XML file into a string, parses it into an in-memory DOM object, queries it using standard DOM methods (getElementsByTagName, createElement), and reserializes it back to disk via XMLSerializer.

3.4 XSLT Transformations

To bridge the gap between strict XML storage and browser-ready presentation dynamically, the system relies on XSLT (Extensible Stylesheet Language Transformations). Stylesheets like messages_to_html.xslt and notifications_report.xslt are processed directly on the server to convert raw XML trees into styled HTML templates, which are then shipped immediately to the React frontend.

3.5 Scripting Integration

TypeScript heavily orchestrates this flow. server.ts maps HTTP API endpoints (like /api/fleet or /api/messages) to file-system operations. The frontend dataService.ts scripts manage fetching these endpoints safely, processing XSLT-induced payloads, parsing JSON responses automatically, and safely mapping them into React component states.

4. Technical Explanation
4.1 How Each Requirement Was Implemented

XML Document Storage: We created a /data directory housing all .xml databases and `.xslt` templates. This replaces traditional relational stores. Endpoints wrap fs.promises to achieve concurrent reads.

XSLT Implementation: Express routes such as /api/chat-history explicitly load messages.xml, load the corresponding styling template, apply the xslt-processor, and return text/html. The frontend then securely injects this via dangerouslySetInnerHTML.

Admin Live Chat: Implemented using Socket.io within the /api/messages POST route and WebSocket listeners. It establishes a live room-based connection where users join their own channel, and admins join the "admins" channel. Broadcasts are emitted to appropriate socket rooms while safely saving messages through the XML DOM parser asynchronously.

Safe Resilience Layer: Implemented a 3-retry HTTP wrapper (while (retries > 0)) inside the frontend data scripts (dataService.ts) to prevent the UI from failing fast on network stutters, granting Node.js time to recover its event loop during high load.

4.2 Challenges Encountered and Solutions
ChallengeSolution
Network Desync / "Failed to fetch" Startup
Frontend attempted to load XML data bound before Express fully attached to the port.		 Built an automatic 3-retry network backoff mechanism inside dataService.ts. If an error occurs, it pauses for 1000ms and tries again, completely stabilizing initial page loads.
Vite WebSocket Disconnections
The HMR WebSocket kept throwing "WebSocket closed without opened" due to misaligned Express routing.		 Architecturally extracted http.createServer(app) in server.ts and explicitly tied Vite's internal development middleware server to the parent HTTP server instance instead of detaching it.
