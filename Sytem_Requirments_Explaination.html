1. Messaging System

The system integrates three core messaging components to fulfill the messaging requirements, utilizing Express backend mechanics to simulate Message Broker and Queue functionality.

1.1 Real-time Chat System (server.ts - Line 221 & 19)

A live support chat interface is implemented via the /api/messages endpoint and Socket.io WebSocket connections. Users communicate dynamically with human admins on the platform in real-time, bridging queries to logistics teams instantly without needing page reloads.

1.2 Task Queue System (server.ts - Lines 164 & 168)

A message queue acts as the system's background task runner. The messageQueue array (line 164) in server.ts offloads incoming API requests over to a background setInterval consumer loop (line 168) (acting like a simulated Kafka worker). It processes messages systematically, broadcasts them via Socket.io to admins and users, prevents overwhelming the server synchronously, and writes them durably to XML storage asynchronously.

1.3 Notification System (server.ts - Lines 500 & 503)

Updates about system maintenance and message resolutions are brokered through the produceNotification() function (line 503). Events are sequentially logged to a notificationQueue (line 500) and flushed by a dedicated consumer process loop (line 514) to notifications_log.xml securely.

2. XML Data Modeling and Usage

To eliminate standard relational database dependency on primitive entities, Rent4Cars builds entirely upon robust, local XML storage ensuring structured hierarchies and integrity.

Where to look: /data/ Directory
  • fleet.xml: Defines <fleet> and <car> tags modeling all rental inventory and specifications.
  • locations.xml: Maps real-world operational hubs inside <locations> utilizing parameters like <city>, <address>, and <lat>/<lng>.
  • messages.xml: Captures multi-directional user chats in <message> components.
  • wishlist_activity.xml: Event logging tool preserving user wishlist action trails wrapped under <wishlist_activities> tag groups.
3. XML Parsing (DOM)

Rent4Cars opts primarily for DOM Parsing (tree-based approach) for reading XML schemas across the application environment.

Implementation Details (server.ts - Lines 107 & 139)

Using the @xmldom/xmldom library, raw XML strings are injected into DOMParser().parseFromString(xmlStr, "text/xml"), converting the text into a fully navigable DOM document tree.

  • To read the document systematically, functions utilize standard selectors: doc.getElementsByTagName("hub").
  • This pattern extracts key parameters sequentially via .textContent directly off the node properties as seen aggressively inside both /api/fleet (line 107) and /api/locations (line 139) GET controllers.
  • For mutations, items use doc.createElement() and re-serialize the entire DOM securely back to the file system format through new XMLSerializer().serializeToString().
4. XSL and XSLT Transformation

Instead of mapping JSON to interface manually via custom logic, XSLT renders complete markup modules instantly directly upon server request.

Engine Used: We utilize the xslt-processor NPM package in server.ts.
4.1 XML to HTML Pipeline (server.ts - Line 690)
  • The primary conversion occurs inside /api/chat-history (line 690).
  • The raw chat log (messages.xml) aligns with its transformation template (messages_to_html.xslt).
  • The engine executes xsltProcess(xml, xslt) dynamically returning styled, class-ready text/html strings capable of instant UI painting without frontend data mapping.
4.2 Report Generation (server.ts - Line 655)
  • The /api/notifications/report pipeline (line 655) filters individual nodes against a specific userId parameter.
  • It recompiles an intermediate XML tree dynamically using filteredDoc.importNode().
  • It then projects that data through a unique XSL template (notifications_report.xslt) yielding a beautiful, isolated report view directly into the web browser interface.
5. Automation using Scripting Languages

To safely maintain the operational health of Rent4Cars without demanding manual administrative supervision, automation scripts have been provisioned.

5.1 Bash Notification Cleanup (/scripts/cleanup_notifications.sh)
  • Purpose: A functional Bash deployment to act as a system monitor/archiver for oversized log arrays.
  • Process: The script systematically analyzes the notifications_log.xml filepath upon activation.
  • Execution: It generates a timestamped safe-state backup directory (`/data/backups/`) and copies the active log over, ensuring preservation of prior XML data points. This script supports background execution via standard chron jobs inside deployment hardware.
  • Setup Automation: We also include a start.bat and system_setup.sh script at the root directory granting instant environmental cross-platform deployments (Windows/Linux) and background service bootstrapping in seconds.
